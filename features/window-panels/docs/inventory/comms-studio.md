## Domain map (the three families in this chunk)

This chunk is not one domain — it's **three families plus two singletons**, and the consolidation story is per-family:

1. **Human messaging** (`messagesWindow`, `singleMessageWindow`) — Supabase user-to-user DMs. Built on `features/messaging/` (`ChatThread` → `useSupabaseMessaging` → `MessageBubble`/`MessageInput`/`OnlineIndicator`, realtime via `postgres_changes`). **This is NOT the agent chat-core** and correctly so — different domain (people, not agents). `messagesWindow` = list+thread; `singleMessageWindow` = one thread, multi-instance. Clean master/detail split.
2. **Agent chat** (`quickChatWindow`) — the lone agent-chat panel here, and it **fully reuses the canonical agent chat-core** (`AgentConversationColumn` + `useAgentLauncher` + `ChatHistorySidebar`, the same primitives `/chat` uses). The one anomaly: it's registered `kind:"window"` but the OverlayController renders it inside a `SidePanelSurface`, **not** a `WindowPanel` — so it is a side panel wearing a window's registry entry.
3. **Voice / transcription studio** (`voicePad`, `voicePadAdvanced`, `aiVoiceWindow`, `transcriptStudioWindow`, + the sibling `transcriptionCleanup` which is in the Notes/Tasks chunk's slice but lives in this family) — all share `voicePadSlice` or the transcript-studio stack. `transcriptStudioWindow` reuses the canonical `StudioView` (same component as the `/transcription/studio` route, DB-backed redux). The voice-pad pair is the classic competing-duplicate question (see verdict below).
4. **Data-fetch utility singletons** (`scraperWindow`, `newsWindow`) — quick-execute panels that fire an API and show results. Scraper is the most surface-mature panel in the chunk (registered surface + agent context menu); News is the least (local `useState`, server action, no redux, no agent wiring).
5. **Context selector** (`dictionarySelectorWindow`) — not a feature-mirror panel; a compact non-blocking picker for the dictionary context on transcription/TTS surfaces, writing to per-surface user-state. The model for "modal replaced by a draggable panel."

### The two headline verdicts

- **`voicePad` vs `voicePadAdvanced` → merge-to-modes (P1).** They are the same component family, same slice (`voicePadSlice`, same `addTranscriptEntry`/`clearAllEntries`/`setDraftText` actions), same `<VoicePadExpanded>` body, same mic. Advanced is literally **Pad + a history sidebar + a richer footer (font-size, new-session→save-to-notes) + a wider default rect**. Every difference Advanced has maps onto a WindowPanel capability the base pad already has the slots for (`sidebar`, `footerLeft/Right`). This is a textbook "Advanced is a *mode* of Pad" — collapse to one `<VoicePad mode="simple"|"advanced">` (or `showHistory`/`showRichFooter` props) driven by the registry's `defaultData`. Keeping two overlay ids + two openers + two tools-grid tiles + two catalogue entries is pure duplication tax.
- **The user wants a voice pad "available everywhere" — it nearly is (Table C).** `voicePad` is already the single best-distributed panel in the chunk: shell sidebar toggle (`SidebarVoicePadToggle`), mobile dock (`MobileDockVoiceButton`), quick-actions hub, BOTH context menus (`UnifiedAgentContextMenu` v2 + `UnifiedContextMenu` v1), the note context menu, the transcripts admin page, and URL hydration. The simple `voicePad` is the everywhere-default; `voicePadAdvanced` is wired only to `DesktopLayout` + the admin page. **The distribution mismatch is itself an argument for the merge** — one pad, distributed once, with Advanced reachable as a mode toggle in the header, so "everywhere" delivers the full pad rather than the stripped one.

## Synthesis (top findings for the master doc)

1. **`voicePad` vs `voicePadAdvanced` = merge-to-modes (P1·M).** Same slice, same body (`VoicePadExpanded`), same mic; Advanced's only deltas (history sidebar, rich footer, wider rect) are WindowPanel slots the base pad already supports. Two overlay ids / openers / tiles / catalogue entries for one capability. Collapse to one mode-driven pad. This also **fixes the "voice pad everywhere" wish**: the everywhere-distributed `voicePad` is the *simple* one, so today "everywhere" ships the stripped pad — one merged pad with an Advanced mode toggle makes the full pad ubiquitous in a single distribution.

2. **Chat-core reuse — clean, by family.** `quickChatWindow` fully reuses the canonical agent chat-core (`AgentConversationColumn` + `useAgentLauncher` + `ChatHistorySidebar`, mirrors `/chat`). `messagesWindow`/`singleMessageWindow` deliberately do NOT — they're human-to-human Supabase DMs (`ChatThread`/`useSupabaseMessaging`/realtime), a justified separate domain, and they share their own core cleanly (list + detail both use `ChatThread`). No forks of the agent core in this chunk.

3. **`quickChatWindow` registry↔render drift (P1·S).** Registered `kind:"window"` but the OverlayController renders it inside a `SidePanelSurface`, not a `WindowPanel`. It also has **no Tools-Grid tile** (a genuine missing tile per PANEL_INVENTORY §6). Reconcile its identity (real window vs declared side panel) and add the tile.

4. **Scraper "grid-only, 0 bespoke call sites" pattern CONFIRMED — and it's the worst-placed of the high-value panels.** `scraperWindow` is the *most* agent-ready panel in the chunk (only one here that's a **registered surface** `matrx-user/scraper`, with full agent context-menu + SurfaceValues + RAG ingest) yet has **zero** bespoke call sites — Tools-Grid only. It should fire from agent-builder / RAG / research / agent-context flows ("pull sample data", "index this page"). Same grid-only gap (less severe) for `aiVoiceWindow`, `messagesWindow`, `newsWindow`.

5. **Surface registration gap (S3).** 9 of 10 panels are NOT registered surfaces — only `scraperWindow` is. `quickChatWindow` registers an internal *agent widget* surface for fork/retry but not a window surface. None of the 10 have the new std header controls (S1 doesn't exist yet). The three already feeding live context to agents (`quickChatWindow`, `transcriptStudioWindow`, `scraperWindow`) are the natural S2 reference implementations.

6. **Orphans / odd ones:** No true orphans in this chunk (every panel has a working opener). Tile gaps: `singleMessageWindow` and `quickChatWindow` have no Tools-Grid tile (singleMessage intentionally; quickChat should get one). `newsWindow` is the weakest panel — local `useState` + server action, no redux/DB/agent/RAG — a demote-or-invest candidate. `aiVoiceWindow`'s state model is opaque (delegated wholly to `AiVoiceFloatingWorkspace`) and it lacks dictionary-context wiring that its STT siblings have.
