## Chunk findings (synthesis)

1. **Content-editor windows â€” already-one at the opener, merge-to-modes at the window.** All three (`contentEditorWindow` / `â€¦ListWindow` / `â€¦WorkspaceWindow`) are dispatched through ONE variant hook (`useOpenContentEditorWindow({variant})`) and built from the same shared official cores (`ContentEditor`, `ContentEditorList`, `ContentEditorTabsWithList`) and the same `useContentEditorEmitter` callback channel. The three window components are thin and nearly identical (single editor â†’ +list sidebar â†’ +tabs). **Verdict: collapse to one mode-driven `ContentEditorWindow(mode: "single"|"list"|"workspace")`** revealing the sidebar/tabs by mode. The callback-group ref pattern here is the cleanest in the chunk and worth keeping as the model. (P1Â·M)

2. **Picklists are a 3-to-4-way competing duplicate â€” the biggest finding (P0).** `udt_picklists`/`udt_picklist_items` are edited by **four** distinct UIs over the same tables: Picklist **V1** (sidebar+spreadsheet, 47 KB), **V2** (compact switcher+flat table, 38 KB, shares `usePicklists` with V1), **V3** (Notion-style, 52 KB, self-contained â€” `/lists/v3`, no window/opener), and **List Manager** (`ListManagerFloatingWorkspace`, its own forked 5s-polling workspace). There is even a `PicklistLanding` page whose job is to make the user *choose between V1/V2/V3* â€” a duplication so visible it shipped as a UX. **Action: owner decision on the ONE canonical picklist core, then delete the other two cores + their `/lists/v*` routes + windows, and re-point List Manager at the canonical core.** `picklistManagerV1Window` + `picklistManagerV2Window` are both MISSING Tools-Grid tiles â€” do NOT add tiles until the canonical core is chosen (adding both perpetuates the duplicate).

3. **`markdownEditorWindow` is a misnamed debug harness, not a duplicate editor.** It wraps `MarkdownClassificationTester` (a dev tool for testing markdown processors/coordinators/configs against sample data); the sibling `markdownEditor` widget wraps the fullscreen variant of the same harness. The real editors are `@/components/official/content-editor/ContentEditor` (embedded multi-mode) and `chat-markdown/FullScreenMarkdownEditor` (production message editor). So it is **not** a duplicate of a real editor â€” but its Tools-Grid tile labeled **"Markdown" in the content category falsely promises a user editor**. Action: relabel + admin/dev-gate the debug window, and point the content "Markdown" tile at a real editor (`ContentEditor` in markdown mode). (P1Â·S) The window+widget pair are also redundant debug variants of each other (P2Â·S dedupe).

4. **Data panels are end-to-end forked (no Redux).** `quickDataWindow`/`quickData`, `listManagerWindow`, and all picklist windows hold state in component `useState` + direct Supabase RPC/service calls â€” none use a Redux slice or `createSelector`, violating the platform "DB â†’ Redux slice â†’ selectors â†’ core" invariant. `listManagerWindow` even **polls every 5s** instead of Redux/realtime. Quick Data is the strongest data panel (full row/cell/column CRUD + version history via `UserTableViewer`) and the best-distributed (~12 bespoke call sites from markdown table/csv/json blocks + data-review) â€” it is the model for *reach*, but should be lifted to a Redux slice. (P1Â·L each)

5. **Quick Data window vs sheet = chrome-only fork.** `quickDataWindow` and the `quickData` sheet render the same `QuickDataSheet` core with different overlay chrome. Same is structurally true of the markdown window vs widget. Collapse each pair to one panel with a presentation prop. (P2Â·S)

6. **Menu gaps & orphans.** No orphans in this chunk â€” **every panel has an opener** in `features/overlays/openers/` (an earlier search missed them because openers live there, not under `features/window-panels/openers/`). MISSING Tools-Grid tiles: `picklistManagerV1Window`, `picklistManagerV2Window` (both flagged in the master doc â€” gated on the canonical decision), `workingDocumentWindow` (correctly absent â€” it needs a `conversationId`, so a context-less grid tile makes no sense), the `quickData` sheet, and the `markdownEditor` widget (redundant). **Zero surface registration and zero std header controls (agents/help) across all 11 panels** â€” `workingDocumentWindow` is the highest-value surface candidate since it is already conversation/agent-bound.
