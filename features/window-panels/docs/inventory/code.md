# Window Panel Inventory — Chunk: Code

> Rows for the 5 code panels: `codeWorkspaceWindow`, `codeEditorWindow`, `codeFileManagerWindow`, `multiFileSmartCodeEditorWindow`, `smartCodeEditorWindow`.
> Column contracts + legend (✓/◑/✗/—; Priority P0-P2; Effort S/M/L) per [`../../PANEL_INVENTORY.md`](../../PANEL_INVENTORY.md).
> Components: `features/window-panels/windows/code/`, `.../smart-code-editor/`, `.../multi-file-smart-code-editor/`, `features/code/host/`.

**Domain family.** All 5 are the **Code** family. Two competing cores: (1) **CodeWorkspace** — a full IDE shell (own ActivityBar + SidePanel + Provider + adapters, route at `/code`); (2) the **Monaco multi-file editor** core shared by CodeEditorWindow + MultiFileSmartCodeEditor (both consume `useCodeEditorWindowState` + `SmallCodeEditor` + `CodeSidebar` + `CodeEditorTabBar`). SmartCodeEditor is a third, 4-column agent-editor shell (`features/code-editor/agent-code-editor/`). CodeFileManager is the file-browser (not an editor).

## Chunk synthesis

1. **Consolidation verdict (key question): NOT 5 separate editors — really 3 shells, with a clear merge.** `codeEditorWindow` + `multiFileSmartCodeEditorWindow` are the **same Monaco editor** (shared `useCodeEditorWindowState`/`SmallCodeEditor`/`CodeSidebar`/`CodeEditorTabBar`) differing only by an attached agent → **merge to one mode-driven editor** (`agentId?` flips agent on). `smartCodeEditorWindow` is a competing 4-column editor+agent answer to reconcile with that merge. `CodeWorkspace` (IDE) and `codeFileManagerWindow` (browser) are legitimately separate.
2. **Orphan: `codeWorkspaceWindow` window form is unreachable.** Registered, render-wired in OverlayController, has a codegen opener — but the opener is **never invoked**, there is **no Tools-Grid tile**, and the only real CodeWorkspace usage is the `/code` route (`CodeWorkspaceRoute`) and `CodeWorkspaceModal`. Either give the window an open path (tile + a launcher) or retire the window form via `remove-window-panel`.
3. **Missing surfaces (S3): all 5.** None of the code panels are registered surfaces in `features/surfaces/`. (`SMART_CODE_EDITOR_SURFACE_KEY="code-editor"` is an agent-LAUNCH surfaceKey, not a window-panel surface — easy to mistake.) Without surface registration, no per-window agent assignment.
4. **Std header controls + help-assistant context (S1/S2): zero coverage.** No panel has the Agents/Help header controls or KV/page-awareness wiring. The two agent editors sync IDE context to the *running agent* (`useIdeContextSync` vsc_* keys) but expose nothing to a help assistant, and the thin `smartCodeEditorWindow`/`codeWorkspaceWindow` shells can't even introspect their own editor state to provide it.
5. **Non-canonical cores + duplicates:** Monaco editor core (canonical, shared by CodeEditor+MultiFileSmart) ✓; competing shells = **CodeWorkspace IDE** (`features/code/`) and **SmartCodeEditor 4-column** (`features/code-editor/agent-code-editor/`). The Monaco twins should collapse; SmartCodeEditor should be reconciled against the merged editor.
6. **Usage gaps / placement:** `smartCodeEditorWindow` is the only well-distributed panel (2 bespoke call sites in CodeBlock + MultiFileCodeEditor). `codeEditorWindow`'s opener hook is invoked by **nobody** — the file-manager bridge bypasses it with a raw `openOverlay` + `require()` lazy import (route it through the opener). `codeFileManagerWindow` and `multiFileSmartCodeEditorWindow` are **Tools-Grid-only** (0 bespoke launchers) — under-surfaced where users edit code. None of the 5 use `urlSync`; `codeFileManagerWindow` has no `onCollectData` so folder/search selection is lost on reload.
