## 2. NOT DONE — the two asks from the last message

### 2a. Make Scribe's first chunk the same size as all others (kill the "combo")

**Why this exists / why it's wrong here:** the recorder gives a couple of tiny
chunks at the very start (rotation `3s, 3s, 4s, then 10s`) for *instant visual
feedback*, then at `idx === 2` it **combines chunks 0–2 into one ~10s "combo"**
and emits that. That combo is the source of the duplicate text the user saw
("According to these findings, According to these findings…") because chunk 0
and the combo both carry chunk 0's words. For a **transcription UI** that 3–5s
head-start is pointless — we should just use one uniform chunk size.

**The constraint (do NOT break this):** the SAME recorder powers the **chat-input
mic** (`features/audio/components/MicrophoneIconButtonCore.tsx`, a *direct*
`useChunkedRecordAndTranscribe` consumer) where the fast first chunk IS wanted.
So the change must be **opt-in**, applied only to the Scribe/Studio path.

**Exact mechanics (already traced — file: `features/audio/hooks/useChunkedRecordAndTranscribe.ts`):**
- The fast schedule lives in `scheduleNextRotationRef` (~lines 653–665): `let delay = 10000; if (currentIdx===1) 3000; ===2 3000; ===3 4000;`.
- The combo lives in `transcribeBlob` (~lines 436–444 build the combo blob when `idx===2`; ~489–500 and ~519–544 emit it with `tStart: t0.tStart`).
- There is already a `chunkDurationMs` prop (line ~81, default `AUDIO_LIMITS.CHUNK_DURATION_MS`) but **the rotation ignores it** — it uses the hardcoded ladder above.

**Recommended fix:** add a boolean option to the hook, e.g. `uniformChunks?: boolean` (or `firstChunkFast?: boolean` default true). When uniform:
1. `scheduleNextRotation` uses a constant `chunkDurationMs` for every rotation (no `3s/3s/4s` ladder).
2. `transcribeBlob` **skips the combo** entirely (treat every chunk normally).
Then thread it through `GlobalRecordingProvider` (the studio/scribe recorder, `providers/GlobalRecordingProvider.tsx:109`) as `uniformChunks: true`. Leave `MicrophoneIconButtonCore` and `useRecordAndTranscribe` on the default (fast) path. Net: Scribe/Studio store clean ~10s chunks with no combo/dup; inputs keep their snappy feedback.

**Note:** the recorder is a *singleton* in `GlobalRecordingProvider`, shared by
**Studio** and **Scribe** (both record via `recording.start({context:{kind:"studio"...}})`).
Studio is the live 4-column workspace and arguably also fine with uniform chunks,
but if it must keep fast feedback, the option has to be set per-`start()` (pass it
through `StartRecordingArgs` → a ref the recorder reads at start) rather than at
the hook level. Simpler first cut: uniform for the whole provider recorder.

### 2b. Let the user choose their own CLEANING agent

The **assistant** agent is already selectable (user pref
`userPreferences.transcription.scribeAssistantAgentId`, surfaced via
`AssistantAgentBar` + `AgentListDropdown`; see `2026-06-12` change-log entry).
The **cleaning** agent is NOT — `cleanRecordingThunk`
(`features/transcript-studio/redux/cleanRecording.thunk.ts`) falls back to
`DEFAULT_CLEANING_SHORTCUT_ID` (`../constants`) and only takes an optional
`shortcutId` override that nothing in Scribe sets.

**Recommended:** mirror the assistant-agent pattern — add a user pref
(`userPreferences.transcription.scribeCleaningAgentId` or a per-session override
on `studio_session_settings`), surface a picker in the session settings sheet,
and pass it into `cleanRecordingThunk({ shortcutId })`. The cleanup feature
(`features/transcription-cleanup/`) already resolves "any agent" via surface
bindings (`useSurfaceAgentRoles`) — reuse that resolution rather than inventing a
new one. `cleanRecordingThunk` already accepts `shortcutId`, so the thunk side is
done; this is a settings-UI + plumbing task.

## 3. OPEN ISSUES reported by the product owner (must be in scope going forward)

1. **Audio behavior when switching tabs is still wrong.** Switching Record /
   Agent / Live / Agent+ does something bad to audio (recording and/or playback).
   Needs careful investigation. Suspects: the tabs all stay mounted
   (`ScribeScreen` keeps all four mounted, only toggling visibility) and the Live
   tab opens its OWN `AudioContext`/track via `features/voice-agent/audio/audioCapture.ts`
   while the shared recorder + `SessionAudioPlayer` (`useScribeSessionAudio`, one
   `HTMLAudioElement`) are also live. Mic-stream handoff (`features/audio/micStream.ts`,
   ref-counted) across tabs is the likely culprit (see KNOWN_DEFECTS D7).

2. **Agent-response audio playback is cut off by a tool call.** When a tool call
   (or the next message) lands mid-response, the current behavior KILLS playback.
   Desired: **let the current spoken response finish, then buffer subsequent
   content and keep playing the full answer** — never read thinking or tool
   output aloud. Today `useAutoVoiceResponse` (`features/transcript-studio/hooks/`)
   drives `useCartesiaStreamingSpeaker` off the live request's accumulated text
   and `stop()`s on `error|cancelled|timeout` and (the bug) on the request
   churn around tool calls. Fix lives in `useAutoVoiceResponse` +
   `useCartesiaStreamingSpeaker` (`features/tts/hooks/`): don't `stop()` on a
   tool-call/status transition; keep a play queue and append the next text run.
   `selectSpokenText` already strips thinking, so the "never read thinking/tools"
   half is done — this is purely about not cutting off + buffering.

3. **The Live tab is completely broken** (xAI Grok realtime voice,
   `ScribeLiveScreen` + `features/voice-agent/`). Recent work added tool-call
   support to it which may have broken it. Needs a dedicated fix pass.

4. **Agent selection on the Agent tab is misleading.** It shows "Select agent"
   with nothing selected, yet a conversation still happens — because there IS an
   agent (the resolved default), we're just not surfacing it. Desired behavior:
   - On load, the tab shows the **already-resolved assistant agent** (no "Select
     agent" empty state). The resolution exists:
     `resolveDefaultAssistantAgentId` (`redux/assistantRoster.ts`) → user pref →
     seeded `AUDIO_ASSISTANT_AGENT_ID`. The agent name just isn't shown because
     the agent definition may not be loaded into the agents slice yet — make the
     bar load/show it.
   - A **placeholder conversation is READY but not yet created** (this now aligns
     with §1's fix: mint is Redux-only until the first turn). Good.
   - **New** button → starts a fresh conversation with the **currently selected**
     assistant. (`switchAssistantAgentThunk({mode:"fresh"})` already does this.)
   - **Related:** conversations aren't reliably associated with the correct
     agent. The roster stores `{conversationId, agentId}` (`assistantRoster.ts`),
     but verify the agentId written at mint matches the one actually used for the
     turn, and that the roster's `agentId` is what the History list reads.

5. **Draft tasks fire with zero confirmation.** In the Agents view, clicking a
   draft task immediately dispatches a call — no confirmation, no explanation of
   what's about to happen, no chance to choose a different action. Add a clear
   pre-execution affordance (what will run, on what, with an approve/cancel).
   Entry point: the draft-task flow is `hooks/useScribeDraftTasks.ts` +
   `AssistantScreen`'s "Draft tasks" action + the UI-first plan/tasks ask-cards
   (`features/agents/ui-first-tools/ui/PendingAsksZone.tsx`, `TaskPanelChip`).
   The click-to-execute lives in those ask-card / task-chip components.

6. **Draft tasks exist on the Agent tab but are MISSING from Agent+.** Invariant:
   **Agent+ must always include everything Agent has, plus more.** The "Draft
   tasks" action was added to `AssistantScreen` (Agent tab) but not
   `ExperimentalAgentScreen` (Agent+). Add it there too (the hook
   `useScribeDraftTasks` is reusable — it just needs the conversationId + send,
   both available in `ExperimentalAgentScreen`).

## 4. KNOWN WEAKNESSES / ARCHITECTURE GOTCHAS (read before touching)

- **The recorder is a chunked-MediaRecorder-rotation singleton**
  (`useChunkedRecordAndTranscribe`, one instance in `GlobalRecordingProvider`).
  Per-chunk `tStart/tEnd` come from `sessionRelativeSec() = (Date.now -
  startTimeRef - pausedDuration)/1000`. The guard `if (!startTimeRef.current)
  return 0` is load-bearing (it's why the 1st recording was the only correct one
  before the fix). Any new code that stamps a chunk time must run AFTER
  `startTimeRef` is set for the current recording.

- **The "combo" chunk (idx 2) DUPLICATES chunk 0's text** — this is the
  fast-feedback artifact §2a kills. Until §2a lands, transcripts show a dup at the
  start of every recording.

- **Two execution paths, two conversation-id semantics:**
  - `executeInstance` (DB-backed; what the Scribe assistant uses via
    `useStudioAssistant.send` AND the chat input via `smartExecute` when
    `apiEndpointMode==="agent"`): Turn 1 sends the client id with `is_new:true`,
    server creates the row under that id, echoes `X-Conversation-ID`.
  - `execute-manual-instance` (Agent Builder, `apiEndpointMode==="manual"`):
    OMITS conversation_id, server mints + echoes its OWN id which is then
    *discarded* (`forceLocalConversationId:true`); history is carried client-side
    in `messages[]`. **Do not point a DB-persisted surface at this path.**

- **Persist conversations ONLY after server confirmation.** The whole §1
  conversation fix rests on this. Never write `assistant_conversation_id` /
  `assistant_conversations` for a conversation that hasn't streamed a turn.

- **`useStudioSession` is shared by Studio AND Scribe.** I deliberately removed an
  earlier `mode` option; if you reintroduce per-surface recording behavior, do it
  via `StartRecordingArgs` (per-`start()`), not a hook-level flag that both
  surfaces inherit.

- **Realtime + supersession is sharp** (`redux/realtimeMiddleware.ts` +
  `slice.ts` `cleanedSegmentApplied/Removed`). Recording-aligned vs time-windowed
  supersession MUST stay in sync between the client reducer and the DB
  `applyCleanupRun` (`studioService.ts`) or cleans vanish. This already bit us
  (see `3b1004670`).

- **Heavy parallel editing happened during this session** — a couple of my edits
  to `useStudioAssistant.ts` / `assistantAgent.thunk.ts` were reverted mid-flight
  by a concurrent save and had to be re-applied. If something looks half-wired,
  diff against the commits in §1.

- **`features/transcripts/hooks/useTranscriptsHub.ts`** had an in-progress
  `recording` hub-kind that left a type error (`Record<TranscriptHubKind,
  SectionState>` missing `recording`). Not mine — owned by whoever is adding the
  recording hub section; confirm it's resolved.

- **`components/admin/server-logs/CoolifyLogViewer.tsx:362`** had a pre-existing
  `string|boolean` → `boolean` TS error (unrelated; a spawned task was filed).
