# War Room — FEATURE.md

> Session-based multitasking command center. Desktop-first. A user opens saved
> **War Rooms**, each a **cockpit of threads**: a **Stage** mode (a live
> watchlist rail + one driven thread) and a **Grid** mode (the self-arranging
> bento gallery, all at once), toggled in the header. Every thread bundles a
> Task + Notes + Audio transcript + Files/Documents + an Agent (the real Scribe
> Agent+ panel) behind six tabs, is context-aware, and can be pinned, parked, or
> projected. **Read this before touching any `features/war-room/**` code.**

**Status:** Active — the cockpit is shipped (Stage⇄Grid, instrument projector, density dial, live meter, kind accent-rails, metric chips, parked-thread rail; Waves 1–8 + the UI bake-off consolidation). **Owner route group:** `(core)`.

---

## What it is

The answer to context-switching: a War Room is a cockpit, not a wall of equal
cards. **Stage mode** keeps every thread alive in a glanceable watchlist rail
(each row shows a live status word + an "is-alive" pulse glyph) beside the one
thread you're driving on a full-height Stage — click any rail row to snap it onto
the Stage with full working state. **Grid mode** is the bird's-eye bento gallery
of every thread at once, arranged and resized by the generic gallery engine
(video-call gallery math + bento at low counts), so 3 tiles look intentional and
12 still hold up. Each thread also carries a **Files** tab — real cloud files
(upload or attach existing) and real editable documents (`udt_documents`) linked
to the tile — and an **Agent** tab: the real Scribe **Agent+** panel (an
assistant conversation + voice in/out + a user-and-agent co-edited working
document), bound to the SAME `studio_sessions` row the Audio tab records into, so
the tile's recordings are the agent's transcript context. The header is mission
control: a **Stage⇄Grid** toggle, the
**instrument projector** (force every thread to one view — all-Tasks, all-Notes…),
a **Comfortable/Compact density dial**, and a live **active / parked / pinned**
meter. Threads pin/park (hide) like Meet participants; parked threads stay
readable as chips and restore-and-stage on click. Rooms are saved sessions you
return to. War Room is a **thin consumer** — it owns the layout engine, the
thread shell, and the session model; all substrate data lives in the existing
features (tasks, notes, transcription, scopes).

## Data model (3 tables, all `ctx_war_room_*`)

- **`ctx_war_room_sessions`** — the room. Owner + multi-scope + `context_scope_ids` (jsonb scope-id array = the session-level context default) + soft-delete.
- **`ctx_war_room_tiles`** — one row per tile. `task_id`/`note_id` FKs (nullable), `context_organization_id`/`context_scope_ids` (NULL = inherit session), `active_tab`, `is_pinned`, `is_hidden`, `position`, **`flavor`** (`thread`|`task`|`project`, default `thread`), **`project_id`** FK → `ctx_projects` (nullable, set-null; for project-flavor tiles — see invariant 9).
- **`ctx_war_room_tile_audio_sessions`** — link table: tile → N `studio_sessions` (`source='war_room'`), `is_active` marks the current one.
- **`ctx_war_room_tile_notes`** — link table: tile → N `notes`, `is_active` marks the current one (the active note also lives on `tile.note_id` for note↔task sync).
- **`ctx_war_room_tile_attachments`** — **polymorphic** link table: tile → N files/documents. `entity_type` ∈ `('user_file','document')` → `entity_id` points at `cld_files.id` or `udt_documents.id`. `UNIQUE(tile_id, entity_type, entity_id)`. The linked entity stays in its own feature; only the link is stored.

RLS follows the `studio_sessions` `check_resource_access(...)` pattern (migration `migrations/ctx_war_room_schema.sql`). Cross-user sharing is not wired yet (register in `shareable_resource_registry` when it is).

## State — the `warRoom` slice (`redux/`)

Stores **only linkage + tile UI state** (sessions/tiles registries, audio links, ephemeral UI). Task data stays in the agent-context tasks slice, note data in `notes`, transcript data in `transcriptStudio`. Selectors are per-key memoized (`redux/selectors.ts`); thunks (`redux/thunks.ts`) are the only writers.

## Room UI (`components/room/` + `components/tile/` + `hooks/`)

The cockpit is one canonical build (the consolidation of a 4-way ui bake-off — winner `reimagine` + grafts from `dense`/`refine`/`sharp`):

- **`WarRoomShell`** — the frame + mission-control header. Routes the body to `StageView` or `WarRoomGallery` by `mode`.
- **`roomViewContext`** — the ONLY home for ephemeral room view state (`mode` stage/grid · `projectedTab` · `density` · the staged thread). React context, **never Redux, never persisted** — these are view preferences, not session data. `DENSITY_LAYOUT` maps the dial to gallery floors; `resolveStagedId` clamps the Stage to a visible thread so it can't strand.
- **Stage mode:** `StageView` (rail + stage) → `RailTile` rows (read-only watchlist, `PulseGlyph` + status word) and `StageTile` (the hero focus pane, full working state). Parked threads fold into a collapsible rail section.
- **Grid mode:** `WarRoomGallery` (the gallery engine + density floors) → `WarRoomTile` (operable card: accent rail + metric chips + segmented tabs + projector; double-click → Stage). Parked threads dock in `HiddenTilesTray`.
- **Shared tile primitives:** `TileTabBar` (segmented, kind-colored switcher, rendered from `TILE_KIND_ORDER`), `TileTabContent` (the 5 real bodies — Task/Notes/Audio/Files/Agent — + combined view; the single source both modes render; the Agent tab is deliberately excluded from the combined "All" view, too rich to stack), `TileMetricChips`, `TileOptionsMenu`, `PulseGlyph`, `ParkedThreadChip`, `tileKind` (semantic accent map aligned with route favicon families: task→success/green, notes→warning/yellow, audio→secondary/purple, files→info/light blue, project stacked section→primary/dark blue, agent→secondary, all→success).
- **Hooks (compose the real slices read-only, written once):** `useTilePulse` (live status word/headline/preview + is-alive signal), `useTileMetrics` (chip readings), `useTileActions` (rename/pin/hide/expand/delete/stage resolver). Stage, Grid, and parked chips all consume these — no forked tile logic.

Every write requires the user's approval (a confirm AskCard) before it applies. Expand via `useOpenTranscriptStudioWindow` (same session as Audio). Nothing reimplemented.
- Context: `EntityTargetPicker` (org) + `EntityScopeTagger` (controlled) via `WarRoomContextPicker`.

## Invariants (load-bearing)

1. **Context is a controlled selection the records carry.** War Room pickers persist org/scope ONLY onto `ctx_war_room_*` rows (`setSessionContextThunk` / `setTileContextOverrideThunk`). They **never** write `appContextSlice` (global active context) or `ctx_scope_assignments`. `selectTileEffectiveContext` resolves `tile override ?? session default`. See [`features/scopes/FEATURE.md`](../scopes/FEATURE.md).
2. **A tile is the container; its tabs are thin consumers.** A tile = `{task_id?, note_id?, audio sessions[], attachments[]}`. Tasks/notes/transcripts/files/documents are owned by their features and survive tile/room deletion — detaching a file/doc removes only the link.
3. **The "new" tile is a render concern**, not a DB row, until the user captures into it.
4. **Room view state is ephemeral.** Stage/Grid mode, the projector, the density dial, and the staged thread live ONLY in `roomViewContext` (React context) — never Redux, never persisted, never on the record. The projector overrides which tab is *shown*, never the tile's saved `active_tab`. Don't reach for a slice for any of these.
5. **Desktop-first.** The gallery math assumes a wide viewport.
6. **The agent edits the tile only through HITL-gated, tile-bound tools.** The `war-room-tools` family (`features/agents/war-room-tools/`) mirrors `features/agents/ui-first-tools/` and is the ONLY way the assistant mutates a tile. Each tool is offered as an **inline** spec (no server-side registry change), **armed per-conversation** by `TileAgentPanel` (`setClientTools` + `registerWarRoomToolBinding`) so it exists ONLY on a War Room tile's Agent conversation, and **resolves its target tile from the binding** — never a guess (no binding ⇒ the dispatcher refuses with `no_tile_bound`). Every write **requires explicit user approval** (a confirm AskCard, reusing the ui-first `pendingAsks` surface) before it runs; on decline the agent gets a non-error "declined" result. Handlers call the **real feature writers** (tasks thunks, `notesApi`, `renameTile`) so Redux + every open surface update live — never a war-room-only mirror. Read `features/agents/ui-first-tools/FEATURE.md` for the client-delegated tool pattern this clones.
7. **The MASTER and ROOM agents act across threads by NOTIFY-AND-WATCH, not approve-each.** The `war-room-master-tools` family (`features/agents/war-room-master-tools/`) is the cross-thread tool set, armed on TWO surfaces: the master agent (`/war-room/all`, full set: `war_room_read_thread`, `war_room_message_thread` (`fresh`|`fork`), `war_room_create_room`, `war_room_rename_room`) and the TIER-2 room agent (in the room shell, the SAME set **minus `war_room_create_room`** — a room agent reads/messages its threads + renames its room but doesn't spin up rooms). Deliberately **NOT** HITL-gated (the per-tile invariant 6 model is the opposite, on purpose): each runs immediately, and messaging a thread fires a toast + opens a live-watch `WindowPanel` (the shared `warRoomWatch` slice → `MasterWatchLayer`, mounted on BOTH surfaces) so the user **sees it stream and steps in**. Safety = Zod-validate args + **resolve the target** (`service/threadResolver.ts` resolves a `thread_id` = tile id → its agent `conversationId`; rooms by id) and refuse cleanly (`ok:false`) on unknown — never guess. The dispatcher + resolver are **room-agnostic**; what scopes the room agent to its room is its single-room ROSTER (`buildRoomAgentContext`), not the tools. Armed per-conversation by `useMasterAgent` / `useRoomAgent` (`setClientTools(...)`, cleared on unmount). `fresh` = `createManualInstance` seeded with `buildTileAgentContextEntries` + the message; `fork` = `forkConversationServer` the thread's existing chain. All from existing primitives — **NO DB, NO server change.**
8. **Three tiers of agent, one widening lens.** A War Room exposes the same agent stack at three scopes, each reusing the canonical `AgentConversationColumn` + a durable conversation + a READ-ONLY roster (never a parallel store): **Tier 1 — per-thread** (`TileAgentPanel` → `useStudioAssistant`, the tile's Agent+ conversation, armed with `war-room-tools` to EDIT that ONE thread, HITL-gated — invariant 6); **Tier 2 — per-room** (`RoomAgentPanel` → `useRoomAgent(sessionId)`, one durable conversation PER ROOM in `localStorage` `war-room:room-agent:<sessionId>`, context `buildRoomAgentContext` = `room_agent_role` + `war_room_threads` (just this room's roster), armed with the master tools MINUS create-room — invariant 7); **Tier 3 — master, all rooms** (`MasterAgentPanel` → `useMasterAgent`, one conversation PER USER, context `buildMasterAgentContext` = `master_role` + `war_room_overview` (every room → its threads), full master tools — invariant 7). Tier 2 is literally Tier 3 narrowed to one session: the context builders share the per-thread shape (`MasterThreadEntry`) and the two agents share the dispatcher/resolver. The widening is roster scope, not new machinery.
9. **A room and its threads never hold CONFLICTING projects.** A tile's `flavor` (`thread`|`task`|`project`) is its render/intent discriminator; a `project` tile binds to a `ctx_projects` row (`tile.project_id`) and its Task tab is that project's task list. A room can ALSO carry a project (`session.project_id`). The invariant: **if the room has a project, every tile is NULL (inherits) or equal to it; per-thread projects (different projects on different threads) are allowed ONLY when the room has none.** Adding/assigning a thread on a DIFFERENT project than the room's raises `ProjectConflictDialog` with the two (and only two) coherent resolutions — **per-thread** (`convertRoomToPerThreadThunk`: stamp the room's project onto every existing tile, THEN clear the room's, give this thread its own) or **keep-room** (this thread joins the room's project). The mirror, "make the whole room one project", is `absorbRoomIntoProjectThunk` (clear differing tile projects + set the room's). Tasks created in any tile auto-associate via the app-wide `ctx_tasks.project_id` (`createTileTask` stamps `selectEffectiveTileProjectId` = `tile.project_id ?? session.project_id`) — War Room is a VIEW onto the project↔task link, never a parallel store. Decision logic is `checkTileProjectConflict` (sync) + `setTileProjectThunk(resolution?)`; the cross-org "pick any of my projects" picker is `WarRoomProjectPicker` (flat over `useUserProjects`, because the canonical `EntityTargetPicker kind="project"` is org-gated). **NO server change** (one idempotent migration: `migrations/ctx_war_room_tiles_flavor_project.sql`).

## Doctrine

- **Build the platform, not the artifact.** The gallery engine and `TaskCommentPopover` were extracted as reusable primitives, not buried here. Before adding a tile capability, check whether the substrate feature already exposes it.
- New routes/panels/overlays/components → add to the `/war-room/admin` map config.
- After substantive changes, update this doc + the Change Log.

## Known deferral

The deeper **transcription working-state Redux migration** (Wave 0) — lifting `useCleanupSession`'s local edit/run state into `transcriptStudio` so the embedded Audio pad and its expand overlay share **live in-progress clean edits** — is deferred. The Audio tab now embeds the full `CleanupPad` pipeline, so the *committed* transcript and the persisted *cleaned* output are both shared via DB + the studio slice; only a clean edit that is still being typed in one view (before its debounce persists) is unsynced to the other. Tracked for when concurrent live editing of the same session across both views is needed.
