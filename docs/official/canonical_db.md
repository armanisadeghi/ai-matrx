# Canonical Entity — Checklist & Access Model

> Confirm on EVERY entity. Legacy pattern → canonical. Objective gate:
> `iam.verify_canonical_ok('<schema>','<table>','<token>')` returns true (target: zero WARN).

## Canonical checklist

**1. Owner** — `created_by uuid` (+ `updated_by uuid`). Kill: `user_id`/`owner_id`/`author_id`/`creator_id` used as owner → `created_by`.

**2. Org** — `organization_id uuid NOT NULL`, FK → `public.organizations`.

**3. Visibility** — `visibility platform.visibility`. Audience axis only (see Access Model). "Make public" = `visibility='public'`. Kill: free-text visibility, or an `is_public` boolean as the access driver.

**4. Sharing (grants)** — grants are ROWS in `public.permissions` (`resource_type`, `resource_id`, `granted_to_user_id`/`granted_to_organization_id`, `permission_level`, `status`, `expires_at`).
- ONE TOKEN: `resource_type` MUST equal the entity token — *identical* across `platform.entity_types.token`, `public.shareable_resource_registry.resource_type`, and `public.permissions.resource_type`. Registry `table_name` is routing only, NEVER the grant token. (Mismatch → `has_access` silently ignores the grant.)
- Register in `shareable_resource_registry` (`owner_column = 'created_by'`).
- Kill: any per-feature `<x>_permissions`/`_shares`/`_collaborators`/`_acl` table or `shared_with` jsonb → migrate rows into `public.permissions`, then graveyard.

**5. Access (RLS)** — generate with `iam.apply_rls('<schema>','<table>','<token>','<variant>')` (NEVER hand-write). Variants: `entity` (owner+org), `component` (defers to parent), `ledger` (append-only, read-only). Emits exactly: `svc_all`, `std_select`, `std_insert`, `std_update`, `std_delete`, plus `pub_read` (anon, `visibility='public'`) when a `visibility` column exists.
- `std_select`/`std_update` LEAD with `created_by = (select auth.uid())` then `iam.has_access('<token>', id, level)` — a `has_access`-only policy breaks `INSERT…RETURNING` (42501).
- Register in `platform.entity_types`. For components, declare the parent in `platform.entity_relationships` (`kind='composition'`, `child_type`, `parent_type`, `fk_column`).
- Kill: bespoke "can-I-see-this" funcs, blanket org-member reads, owner-only checks hard-wired to `user_id`.

**6. Versions** — `version int` bumped by the `_touch` trigger; set `is_versioned=true` in `entity_types`; install the `_history` trigger (`platform._version_capture('<token>')`) → snapshots into `history.row_versions`. Kill: parallel/ad-hoc per-table version tables.

**7. Soft delete** — `deleted_at timestamptz` (NULL = live). Kill: `is_deleted`/`deleted` boolean, `status='deleted'`.

**8. Timestamps & triggers** — `created_at`, `updated_at`; canonical trigger trio `_stamp` (`platform._stamp_actor`), `_touch` (`platform._touch_row`), `_history` (`platform._version_capture('<token>')`).

**One-line test:** "`created_by` + `deleted_at` + `visibility` enum, shares via `public.permissions` keyed on the entity token, reads via `iam.has_access` generated by `apply_rls`, history via `_history` → `row_versions`? If no to any → fix it."

# Access Model

Access is **three independent axes**. The classic "private / internal / discoverable / public"
labels are *combinations* of these axes — not a single scale. Keeping them separate is what
prevents one overloaded enum and lets each primitive do one job.

## Axis 1 — Audience (`visibility`): "Who is allowed in?"
The `platform.visibility` enum, low → high openness:

- **`private`** — Only `created_by`, plus anyone with an explicit `permissions` grant.
- **`internal`** — All members of the entity's `organization_id`.
- **`public`** — Anyone (anon included).
- **`link`** — *Not an audience.* See Axis 3 / grants below; a link is a grant mechanism, not a tier.

> `apply_rls` already enforces this axis: org members via `has_access`/`has_org_access`,
> anon `pub_read` when `visibility='public'`.

## Axis 2 — Listed (`is_listed`): "Does it surface in discovery/search?"
A standalone boolean, **independent of audience** — `entity_types.is_listed` is the default;
a per-row `is_listed` (+ optional `listed_at`) is the override.

- Listing is scoped to whatever the audience already allows:
  - `internal` + listed → appears on the **org's** aggregation/search pages (e.g. the company Agents page).
  - `public` + listed → appears on **public** aggregation/search pages.
  - `private` + listed → findable only by the owner/grantees (rarely useful, but legal).
- Listing NEVER widens audience. An unlisted public item is reachable by link/grant but
  not surfaced in search; a listed internal item is searchable *only within the org*.

## Axis 3 — Projection: "How much does a non-owner see?"
What a viewer receives is the **card** (public-safe columns) vs. the **full row** (owner/grantee).
This is the `agent.card` pattern generalized — declared per entity, not hand-rolled per feature.

- **Card** — safe public face: name, description, logo, creator, etc. Drives search results
  and aggregation tiles.
- **Executor** — the "do something with it" surface (chat with the agent, open the applet, run
  the tool). Gated by `has_access(token, id, 'viewer')`; uses the entity server-side without
  exposing its body.
- **Denied/secret body** — sensitive columns (system prompts, tools, credentials, model config)
  never leave the server for non-owners, regardless of listing or audience.

## Permission levels (what a grant conveys)
`permissions.permission_level`, used by grants and by share-links:
- **Viewer** — view only.
- **Commenter** — view + comment; no content changes.
- **Editor** — create, edit, delete, comment; no ownership/permission management.
- **Owner** — full control: permissions, sharing, ownership transfer, deletion, administration.
  Ownership may be a user or an organization.

## Share-links
A link is a `permissions` row whose `granted_to` is a link token instead of a user/org, carrying
a `permission_level` (`viewer`…`owner`). "Transfer-on-claim" = a link granting `owner`.
A link can attach to an entity at **any** audience tier — it's orthogonal to `visibility`.

## Special handling (NOT a visibility tier)
- **Secret / Protected** — sensitive values needing extra safeguards: API keys, GitHub/OAuth
  tokens, env vars, encryption keys. Handled by the secrets layer, not `visibility`.

## The classic labels, decoded
| Label | Audience (`visibility`) | Listed (`is_listed`) | Non-owner projection |
|---|---|---|---|
| Private | `private` | usually false | none (owner/grantees only) |
| Internal | `internal` | optional (org pages) | card or full, per entity |
| Discoverable | `internal` **or** `public` | **true** | **card** (executor on click; secret body withheld) |
| Public (full) | `public` | optional | full row |

> Visibility answers **who can access**. Listed answers **whether it's surfaced in discovery**
> (within whatever audience allows). Projection answers **how much a non-owner sees**.
> Secrets are an additional protection layer, not another visibility tier.
