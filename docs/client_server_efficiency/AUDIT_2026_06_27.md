# Client ↔ Server Efficiency Audit — 2026-06-27

Initiative: stop the Next.js client from straining the AI Dream (Python) backend and
from making redundant requests. Five rules (from the user):

1. Pure data CRUD never goes through Python — use supabase-js directly (best on edge devices).
2. Python requests only for things Python *processes* (logic/handling) or file read/write of **non-public** content — never pure CRUD.
3. Don't poll the server — use webhooks / Supabase Realtime (fully available, mostly unused).
4. Hunt for the client making multiple requests / habits that strain the server.
5. Eliminate request inefficiencies.

## The canonical example (the probe that started this)

A single click to preview a file (`/files/all/{folder}?file={id}`) fired **5** Python requests:
`/asset`, `/url`, `/analysis`, `/annotations`, `/rag-status`. The user only wanted to *see* the file.

Root cause was three overlapping anti-patterns, now the taxonomy for the whole sweep:

| # | Anti-pattern | Fix shape | Reference |
|---|---|---|---|
| 1 | **Eager-mount fan-out** — a surface mounts many data-fetching children at once, firing requests for things never viewed | lazy-mount-then-keep-alive | `FileTabsBody.tsx` (shipped) |
| 2 | **Python-as-DB-proxy** — FE calls Python REST for a pure RLS-readable DB read | convert to direct supabase-js | `files/api/document-lookup.ts` |
| 3 | **Redundant / duplicate / remount requests** — same data fetched ≥2× per action | once-guard / lift state / module dedup cache | `useFileAsset.ts` module cache |
| 4 | **Polling the server** — timer re-hits Python/Supabase instead of Realtime | wire Realtime / drop interval | `rag-job-realtime.ts` |

## CRITICAL architectural facts (verified live — do NOT re-derive, do NOT assume otherwise)

- **Non-public schemas ARE PostgREST-exposed.** The 2026 reorg moved file tables into a `files`
  schema (`files.analysis`, `files.page_annotations`, `files.file_rag_jobs`, `files.pages`, …).
  The FE already does **456** direct `.schema("files"|"chat"|"agent"|"tool"|"skill"|"workspace"|"app")`
  reads. So Python-as-DB-proxy conversions are valid — but they must use `supabase.schema("files")`,
  **not** `public`. A blind "read the public table" conversion will fail.
- **`rag` schema is NOT exposed** — `rag.kg_chunks`, `rag.kg_entities`, `rag.data_stores`, etc. stay
  server-mediated until/unless `rag` is added to the exposed list. (One high-leverage infra change
  would unlock a whole second wave.) `chunk_count` is therefore left `null` on direct reads (already
  the established behavior in `document-lookup.ts`).
- **Verify impact PER CALLSITE, never from the wrapper's existence.** Many `features/files/api/*.ts`
  wrappers (`listFiles`, `getFile`, `searchFiles`, `listFolders`, …) exist for the **extension /
  backend-mediated** contexts and have **zero or few live FE callers** — the FE list/tree/RAG-column
  paths already read Supabase directly (e.g. the file-list RAG column uses `lookupFileDocument`
  batched+deduped, NOT `/rag-status`). The audits below estimate impact by wrapper; confirm the real
  FE caller and its frequency before converting.
- After the FileTabsBody fix, `/analysis` `/annotations` `/rag-status` only fire when the user opens
  the Analysis / Knowledge / Info tab — so converting them is real but lower-frequency than it looks.
